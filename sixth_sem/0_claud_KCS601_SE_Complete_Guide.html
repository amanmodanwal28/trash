Introduction to Software Engineering

Software characteristics, SDLC models, Crisis — Exam mein sabse zyada aata hai ⭐⭐

What is Software Engineering?
💡 Simple bhasha mein: Software banane ka systematic tarika
📖 DEFINITION (IEEE) Software Engineering is the application of a systematic, disciplined, and quantifiable approach to the development, operation, and maintenance of software.

Iska matlab hai — software ko sirf "code likhna" nahi hai. Isme planning, designing, testing, aur maintenance sab included hain. Jaise civil engineers bridge banate time proper calculation karte hain, waise hi software engineers bhi software banate time proper process follow karte hain.

🔑 KEY COMPONENTS OF SE
  • Process: Software development kaise karna hai — steps & methods
  • Methods: Technical techniques for analysis, design, coding, testing
  • Tools: CASE tools, IDEs, debuggers — kaam ko automate karne ke liye
  • Quality Focus: Har step pe quality ensure karna

Software Components & Characteristics
💡 Hardware aur Software mein kya fark hai — ye samajhna zaroori hai
📦 SOFTWARE COMPONENTS
  • Programs: Executable code (instructions)
  • Data Structures: Data ko organize karne ka tarika
  • Documents: User manuals, technical docs, SRS

Software Characteristics (Hardware se alag kaise hai):

1. Software Deteriorates, Doesn't Wear Out

Hardware physically ghis jaata hai (bathtub curve). Software mein physical wear nahi hota, lekin bugs aur environment changes se "deteriorate" hota hai. Isko "wearout" nahi kehte, "change induced failure" kehte hain.

2. Software is Not Manufactured

Software "build" hota hai, "manufacture" nahi. Matlab assembly line pe nahi banta. Har software ek alag intellectual effort hai.

3. Software is Complex

Modern software mein millions of lines of code hoti hai. Complexity manage karna ek bada challenge hai.

4. Software is Custom-Built

Zyada tar software specific needs ke liye banaya jata hai (custom-built), ready-made components pe depend nahi karta (though reuse badhta ja raha hai).

5. Software Doesn't Wear Out (No Spare Parts)

Broken part replace karne jaisi cheez software mein nahi hoti. Bug fix karna ek naya change hai jo naye bugs introduce kar sakta hai.

6. Software is Flexible

Software ko change karna hardware se asaan hai, lekin poorly designed software mein changes mushkil aur expensive ho jaate hain.

Software Crisis — Kya tha aur kyon hua?
💡 1960s-70s mein software projects fail hone lage — yahi software crisis hai
📖 DEFINITION Software crisis refers to a set of problems encountered in the development of software in the late 1960s. The demand for software increased drastically, but the ability to develop software did not keep pace.

Problems/Symptoms of Software Crisis:

  • Cost Overrun: Budget se zyada kharch ho jaata tha. Projects over budget rehte the.
  • Schedule Overrun: Deadline miss hoti thi. Projects late deliver hote the.
  • Poor Quality: Delivered software unreliable, inconsistent hota tha.
  • Poor Maintainability: Ek baar deliver hone ke baad maintain karna mushkil tha.
  • Increasing Complexity: Hardware fast advance ho raha tha, software piche reh jaata tha.
  • No Documentation: Proper documentation nahi hoti thi.
  • User Dissatisfaction: User ki actual needs poori nahi hoti thi.
💡 CAUSES of Software Crisis
  • Increasing complexity of software problems
  • Lack of adequate training in software engineering
  • Failure to apply proper methods and techniques
  • No systematic approach to development
  • Increasing demand for software (hardware became cheap)

Solution: Software Engineering discipline emerged as a solution — systematic, disciplined approach to software development.

⚠️ 2022 PAPER — Q3(a)

Explain software characteristics in detail. Discuss the reasons of software crisis.
Focus: 5-6 characteristics + 5-6 causes of crisis = full marks

Evolutionary & Agile Models
💡 Iterative Enhancement, Agile, Scrum — modern development ka basis

Iterative Enhancement Model

Pehle simple version banao, phir iteration ke baad features add karte jao. Har iteration mein: plan → design → code → test.

Use when: Large project, need early delivery

Evolutionary Development

Concurrent development of multiple product versions. 2 types: Exploratory (user ke saath work out karo) aur Throwaway Prototyping (poorly understood parts ko explore karo).

Agile Model

Iterative development + collaboration + responding to change. 4 values: Individuals over processes, Working software over docs, Customer collaboration over contracts, Responding to change over plan.

Scrum Framework

Sprint (2-4 week iterations), Daily Scrum, Sprint Planning, Sprint Review. Roles: Product Owner, Scrum Master, Dev Team. Agile ka most popular implementation.

1. ELICITATION (Requirements Gathering) Techniques:

Interviews

Stakeholders se direct questions. Structured (fixed Q's) ya Unstructured (open-ended). Most common technique.

Questionnaires/Surveys

Written questions distribute karna. Large group se data gather karne ke liye useful. Time efficient.

Observation/Ethnography

User ko kaam karte dekhna (without disturbing). Hidden requirements discover hoti hain.

JAD Sessions

Joint Application Development — users, managers, developers saath milke requirements discuss karte hain.

Prototyping

Working model dikhao → user ko requirements clearly samajh aati hain aur feedback deta hai.

Brainstorming

Group session mein ideas generate karo. No criticism during session. Creative requirements nikalni.

SRS Document — Software Requirements Specification
💡 SRS = Software ka "constitution" — sab kuch yahan likha hota hai
📖 DEFINITION SRS (Software Requirements Specification) is a complete description of the behavior of a system to be developed. It describes what the software will do and how it will be expected to perform.

Need of SRS (Kyun zaroori hai):

  • Developer aur customer ke beech agreement document
  • Project planning, estimation ke liye base
  • Testing ke liye benchmark — test cases SRS se derive hote hain
  • Future maintenance ke liye reference
  • Reduces ambiguity aur misunderstandings

Characteristics of Good SRS:

Complete

Saare requirements clearly mention honi chahiye. Kuch bhi miss nahi hona chahiye.

Consistent

Koi do requirements ek doosre ke contradict na karein. No conflicts.

Unambiguous

Har requirement ka sirf ek hi interpretation hona chahiye. Clear language use karna.

Verifiable

Har requirement testable honi chahiye. Prove kiya ja sake ki fulfilled hui ya nahi.

Traceable

Har requirement ko design aur code se link kiya ja sake.

Modifiable

Future changes easily accommodate ho sakein SRS mein.

Software Quality Assurance (SQA)
💡 Quality ensure karna — sirf testing nahi, poora process quality focus hai
📖 DEFINITION SQA is a set of activities for ensuring quality in software engineering processes. It focuses on the process (not just the product) to ensure that quality software is built.

Verification vs Validation

Verification: "Are we building the product right?" — Process check. Are we following standards?

Validation: "Are we building the right product?" — Product check. Does it meet user needs?

Simple: Verification = internal process check. Validation = external user satisfaction check.

SQA Activities

  • SQA Plan preparation
  • Formal Technical Reviews (FTR)
  • Multi-library coding standards
  • Process monitoring and controlling
  • Testing strategies and procedures
  • Change control management
  • SQA record keeping

SQA Framework consists of:

  • Standards: IEEE, ISO standards for software processes
  • Reviews and Audits: Formal technical reviews, walkthroughs
  • Testing: Unit, integration, system testing
  • Error collection and analysis: Track defects systematically
  • Change management: Control changes systematically

Software Design & Metrics

Basic Concepts of Software Design
💡 Design = blueprint banana, jaise architect ghar ka map banata hai
📖 SOFTWARE DESIGN Software design is the process of defining the architecture, components, modules, interfaces, and data for a system to satisfy specified requirements. Design transforms requirements into a blueprint for constructing software.

Design Framework — Roger Pressman ke anusar:

Architectural Design (High-Level)

System ka overall structure. Major components/subsystems aur unke relationships. External interfaces. "What" — kya components honge.

Low-Level Design (Detailed)

Component/module ke andar ki details. Algorithms, data structures, interfaces. "How" — har component kaise kaam karega.

Principles of Good Software Design:

  • Abstraction: Complex details ko hide karo, sirf essential information dikhao
  • Modularity: System ko small, manageable modules mein divide karo
  • Information Hiding: Module ki internal details doosre modules se hide karo
  • Stepwise Refinement: Top se bottom level tak gradually detail karo
  • High Cohesion: Ek module mein related functions rakhna
  • Low Coupling: Modules ke beech minimum dependence
  • Simplicity: Simple design prefer karo — complex se bachna

Design Tools:

Coupling & Cohesion — Most Important Topic!
💡 Coupling = modules ka inter-dependence | Cohesion = ek module ke andar relatedness
🎯 GOAL: Low Coupling + High Cohesion = Good Design Low coupling matlab modules ek doosre pe kam depend karein. High cohesion matlab ek module ke andar sari cheezein ek kaam se related hon.

Software Metrics — Size aur Complexity Measure karna

LOC (Lines of Code)

Code ki lines count karo. Simple but language-dependent. Problems: Comments, blank lines, style differences affect count.

Advantages: Easy to measure
Disadvantages: Language dependent, not a good quality measure

Function Point (FP)

Language-independent size measure. User-visible functionality count karo.

Components: External Inputs, External Outputs, External Inquiries, Internal Logic Files, External Interface Files

Function Point Calculation:

FP = UFC × (0.65 + 0.01 × ΣFi) Where: UFC = Unadjusted Function Count = Σ(count × weight) for each component type Fi = 14 General System Characteristics (0-5 each) UFC Weights: External Inputs: Simple=3, Average=4, Complex=6 External Outputs: Simple=4, Average=5, Complex=7 External Inquiries: Simple=3, Average=4, Complex=6 Internal Logical Files: Simple=7, Average=10, Complex=15 External Interface Files: Simple=5, Average=7, Complex=10
📐 HALSTEAD'S SOFTWARE SCIENCE Measures based on operators and operands in code.
  • n1 = distinct operators, n2 = distinct operands
  • N1 = total operators, N2 = total operands
  • Vocabulary: n = n1 + n2
  • Length: N = N1 + N2
  • Volume: V = N × log₂(n)
  • Difficulty: D = (n1/2) × (N2/n2)
  • Effort: E = D × V
Cyclomatic Complexity — Code ki complexity measure karna
💡 McCabe ne 1976 mein diya — code mein independent paths kitne hain
📖 DEFINITION Cyclomatic Complexity is a quantitative measure of the number of linearly independent paths through a program's source code. Higher value = more complex = more testing needed.

3 Methods to Calculate:

Method 1: V(G) = E - N + 2 Where E = number of edges, N = number of nodes Method 2: V(G) = Number of regions in control flow graph (Count enclosed areas + 1 for outside region) Method 3: V(G) = P + 1 Where P = number of predicate/decision nodes (nodes with more than one exit)

Example from 2022 Paper (Q6a):

Code: IF A = 100 THENIF B > C THEN A = B ELSE A = C ENDIF ENDIF PRINT A Control Flow Graph Nodes: 1: Start + IF A=100 2: IF B>C 3: A = B 4: A = C 5: PRINT A + End Edges: 1→2, 1→5, 2→3, 2→4, 3→5, 4→5 Method 1: V(G) = E - N + 2 = 6 - 5 + 2 = 3 Method 3: P = 2 (two IF conditions) → V(G) = 2 + 1 = 3 Independent Paths: Path 1: 1→5 (A≠100) Path 2: 1→2→3→5 (A=100, B>C) Path 3: 1→2→4→5 (A=100, B≤C) Cyclomatic Complexity = 3

11-20MediumModerate complexity

>50Very HighUntestable, refactor karo

🔑 STUBS vs DRIVERS

Stub: Ek dummy module jo called module ka role play karta hai. Top-Down integration mein use hota hai. Lower level modules ke liye substitute.

Driver: Ek dummy module jo calling module ka role play karta hai. Bottom-Up integration mein use hota hai. Upper level modules ke liye substitute.

🔑 ALPHA vs BETA TESTING

Alpha Testing: Developer ke site pe end users ke saath testing. Controlled environment. Developer present hota hai bugs fix ke liye.

Beta Testing: Customers ki sites pe, real environment mein. Developer present nahi. Customer real usage mein bugs report karta hai. Release se pehle ka final testing.

Black Box vs White Box Testing

⬛ Black Box Testing (Functional)

Internal code ka knowledge nahi — sirf input/output test karte hain. "What it does" test karna. Also called Behavioral testing.

  • Equivalence Partitioning: Input domain ko classes mein divide karo, har class se ek test case
  • Boundary Value Analysis: Boundary values (edges) pe test karo
  • Decision Table Testing: Conditions aur actions table mein
  • Cause-Effect Graphing

⬜ White Box Testing (Structural)

Internal code structure ka knowledge hai. "How it does" test karna. Also called Glass Box/Clear Box testing.

  • Statement Coverage: Har statement kam se kam ek baar execute ho
  • Branch Coverage: Har branch (if/else) test ho
  • Path Coverage: Har independent path test ho
  • Condition Coverage: Har condition T aur F dono ho

COCOMO Model — Cost Estimation
💡 Barry Boehm ka model — LOC se effort aur time estimate karo
📖 COCOMO (COnstructive COst MOdel) COCOMO is an algorithmic software cost estimation model developed by Barry Boehm in 1981. It uses a regression formula with parameters derived from historical project data.

COCOMO Types:

1. Basic COCOMO

Simple, static formula. LOC se effort aur time calculate.

E = a × (KLOC)^b
D = c × E^d

2. Intermediate COCOMO

15 cost driver attributes add karta hai Basic mein. More accurate estimation.

3. Detailed COCOMO

Project ke har phase/module ke liye separate estimation. Most accurate.

COCOMO II (1995)

Boehm ka updated model. Modern development methods support karta hai (OO, reuse, etc.).

Basic COCOMO Formulas:

Project Types: 1. Organic — Small, experienced team, familiar problem 2. Semi-detached — Medium team, mixed experience 3. Embedded — Complex constraints, strict requirements Effort (E) in Person-Months: Organic: E = 2.4 × (KLOC)^1.05 Semi-detached: E = 3.0 × (KLOC)^1.12 Embedded: E = 3.6 × (KLOC)^1.20 Duration (D) in Months: Organic: D = 2.5 × E^0.38 Semi-detached: D = 2.5 × E^0.35 Embedded: D = 2.5 × E^0.32 Staff (People) = E / D Person Month (PM) = 152 working hours (approx)
⭐ EXAM EXAMPLE

Q: KLOC = 10, Organic project. Find Effort, Duration, Staff.
E = 2.4 × (10)^1.05 = 2.4 × 11.22 = 26.93 PM
D = 2.5 × (26.93)^0.38 = 2.5 × 4.04 ≈ 10.1 months
Staff = 26.93/10.1 ≈ 2.67 ≈ 3 people

Q4(b) Compare SEI CMM Model and ISO 9000. Discuss 5 levels of CMM | 10 marks

SEI CMM Model:
CMM (Capability Maturity Model) SEI Carnegie Mellon ne develop kiya. Software process maturity 5 levels mein measure karta hai.

Level 4 — MANAGED: Quantitative management. Process aur product quality statistically measured. Predictions reliable hain. SPC (Statistical Process Control) use hoti hai.

Level 5 — OPTIMIZING: Continuous process improvement. Defect prevention. Technology change management. Innovation culture. Entire organization learning.

• CMM = 5 maturity levels | ISO = Certification (yes/no)

Q5(a) Explain Cohesion and Coupling — all forms | 10 marks
MODEL ANSWER:
Coupling (Modules ke beech dependence — Low chahiye):

1. Data Coupling (Best): Sirf simple data parameters share karte hain. Example: findMax(int a, int b) — sirf 2 integers pass kiye.
2. Stamp Coupling: Poori data structure pass ki jati hai. Example: passing whole student object jab sirf roll number chahiye tha.
3. Control Coupling: Control flags pass karna. Example: processRecord(record, printFlag) — flag se behavior control.
4. External Coupling: External medium/format pe depend karna. Same file format ya global data structure use karna.
5. Common Coupling: Global variables share karna. Multiple modules same global variable modify karte hain. Dangerous!
6. Content Coupling (Worst): Ek module doosre ke internal data/code directly access karta hai. GOTO to another module.

Cohesion (Module ke andar relatedness — High chahiye):

1. Functional (Best): Sirf ek specific well-defined task. "Calculate Interest" = sirf yahi kaam.
2. Sequential: Elements sequential hain, ek ka output doosre ka input. "Read file → Parse → Process"
3. Communicational: Same data pe operate karna. "Update student record" — name, marks, attendance sab same student ka.
4. Procedural: Ek process ke sequential steps. Open file → Read → Close.
5. Temporal: Same time pe execute hona. All initialization functions ek module mein.
6. Logical: Same category lekin unrelated. "All error handling routines" — logical group but different purposes.
7. Coincidental (Worst): Random grouping, koi logical connection nahi. Utility module jisme random functions hain.

(f) Software Architecture: System ka high-level structure — components, relationships, external interactions define karta hai.

(g) Verification vs Validation: Verification = "Building product right?" (internal process). Validation = "Building right product?" (meets user needs). V&V both essential.

(h) Alpha vs Beta Testing: Alpha = developer site pe real users test karte hain. Beta = customer sites pe real environment mein testing, developer absent.

(i) CASE Tools Advantages: Automation of tedious tasks, consistency maintained, documentation automatic, errors reduce.

(j) Adaptive Maintenance: Changing environment ke liye software modify karna. Example: Windows XP ke software ko Windows 11 pe compatible banana.

Iterative Waterfall Model:
Classic waterfall mein phases strictly sequential hain. Iterative waterfall mein har phase previous phase ko feedback de sakti hai (feedback loops).

Phases aur activities:

Iterative feature: Phase 3 problem milne par Phase 2 pe wapas, Phase 4 problem par Phase 3 pe wapas.

SEI-CMM Model: Upar diye gaye Unit 2 notes se 5 levels detail mein likhna.

CMM Level 1 organizations "fire fighting" mode mein hoti hain. Matlab:

Q6(a) Cyclomatic Complexity — Control Flow Graph for given code | 10 marks
MODEL ANSWER:
Given Code:
IF A = 100
  THENIF B > C
    THEN A = B
    ELSE A = C
  ENDIF
ENDIF
PRINT A

Control Flow Graph Nodes:
Node 1: Start
Node 2: IF A = 100 (decision node)
Node 3: IF B > C (decision node)
Node 4: A = B
Node 5: A = C
Node 6: PRINT A + End

Edges:
1→2, 2→3 (A=100), 2→6 (A≠100), 3→4 (B>C), 3→5 (B≤C), 4→6, 5→6
E=7, N=6

Method 1: V(G) = E - N + 2 = 7 - 6 + 2 = 3
Method 2: Regions = 3 (2 enclosed + 1 outside) = 3
Method 3: P = 2 (two decision nodes) → V(G) = 2 + 1 = 3

Independent Paths:
Path 1: 1→2→6 (A≠100, directly PRINT)
Path 2: 1→2→3→4→6 (A=100, B>C, A=B)
Path 3: 1→2→3→5→6 (A=100, B≤C, A=C)

DFD — Data Flow Diagrams Levels | 10 marks
MODEL ANSWER:
DFD (Data Flow Diagram): System mein data kaise flow karta hai uska graphical representation.

Symbols: Rectangle (External Entity) + Oval (Process) + Open Rectangle (Data Store) + Arrow (Data Flow)

Level 0 (Context Diagram): Poora system ek oval process. Sirf external entities aur main flows.
Level 1: System ko 3-7 major processes mein break down karo.
Level 2: Level 1 processes ko further detail mein break karo.

Rules for DFD:
• Process ke liye noun + verb (e.g., "Process Order")
• Data flow ke liye noun (e.g., "Order Details")
• No direct External Entity to External Entity flow
• No direct Data Store to Data Store flow
• Process ke inputs → outputs hongi
COCOMO Model for Project Estimation + Scheduling | 10 marks
MODEL ANSWER:
COCOMO notes Unit 5 mein dekho. Ye numerical example zaroor practice karo:

Problem: 25 KLOC semi-detached project. Effort, Duration, Staff calculate karo.
E = 3.0 × (25)^1.12 = 3.0 × 37.07 = 111.2 PM
D = 2.5 × (111.2)^0.35 = 2.5 × 6.1 = 15.25 months
Staff = 111.2 / 15.25 ≈ 7.3 ≈ 8 people

Project Scheduling:
Gantt Chart: Bar chart showing tasks vs timeline
PERT Chart: Network diagram showing task dependencies
CPM: Critical Path Method — longest path = project duration
Critical Path: Path jisme koi slack nahi — delay this = delay project
